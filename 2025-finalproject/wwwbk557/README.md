# 课程学习总结

在这门课程中，我掌握了 Git 与 GitHub 的联动逻辑，尤其通过  Pull Request（PR）  流程，理解了团队开发中代码提交、评审与合并的完整链路。以下是我的学习过程与核心收获总结。

## 一、从 Git 本地操作建立基础

最初，我先理解了 Git 的“本地仓库”核心概念。它能记录每一次修改并支持回溯，是一个开发者必不可少的工具。课程中我学习到了本地命令，也是后续与 GitHub 联动的基础：

1. 提交代码的“三步走”流程
   每次修改代码后，必须通过固定步骤将变更“暂存→记录→推送”，确保本地与远程同步：

-  git add [文件名] ：将修改的文件加入“暂存区”（相当于“确认要提交的内容”），若要全部暂存则用  git add . ；
-  git commit -m 'Add [文件名]' ：将暂存区内容提交到“本地仓库”，并通过  -m  写清晰的提交说明（课程强调说明需开头大写，必须英文 ）；
-  git push：推送到远程仓库；



## 二、配置SSH密钥

1. 使用命令生成密钥：

   ```bash
   ssh-keygen -t rsa -C "youremail@example.com"
   ```

2. 将公钥配置到GitHub：

- 登录 GitHub → 右上角头像 → Settings
- 左侧菜单 → SSH and GPG keys → New SSH key
- Key 框粘贴刚才复制的公钥 → 点击 Add SSH key

## 二、GitHub Pull Request 协作流程

通过课程我成功连接上了pull request并进行作业提交

1. 先fork到自己的仓库
2. 通过以下指令克隆到自己本地

```bash
git clone git@github.com:用户名/仓库名
```

3. 建立文件夹写入自己的内容
4. 发起 PR：提交代码评审请求

5. PR 审核与修改：配合检查项完善自己内容

- 这一步让我明白：PR 不是“直接合并代码”，而是“请求团队审核我的代码”，是协作中“质量把关”的关键环节。

## 三、关键认知：Git 与 GitHub 的协作逻辑

课程最让我收获的，不是单纯记住命令，而是理解了 Git 与 GitHub 如何“分工配合”：

- Git 负责“本地管理”：记录修改、创建分支、解决冲突，确保每个开发者在本地能安全地迭代代码；
- GitHub 负责“远程协作”：通过远程仓库存储代码、用 PR 实现评审、用 Issues 跟踪任务，让多人能同步推进同一个项目；

## 五、总结：从“会用命令”到“理解协作”

这门课程让我从“完全不懂版本控制”，到理解“本地开发→远程推送→PR 评审→代码合并”的全流程。最大的收获不是记住了多少 Git 命令，而是理解了 “版本控制的本质是协作效率与代码安全”——Git 保障了个人开发的灵活性，GitHub 则通过 PR 等功能，让多人协作时的代码质量可控、流程可追溯。
